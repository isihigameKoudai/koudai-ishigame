```markdown

tags: [技術名] [フレームワーク名] [カテゴリ] フロントエンド
author: isihigameKoudai

---

## 概要
[自己紹介 + 記事の背景・動機]

## [技術/概念の説明セクション]
[基礎知識の解説]

## [本題/実装セクション]
[具体的な手順・コード]

## まとめ
[学びのポイント・感想]

## 参考
[参考リンク]
```

## 文体・トーンのガイドライン

### 自己紹介パターン（冒頭）

**Qiita向け:**
```
こんにちは、都内でフロントエンドエンジニアをしてます[かめぽん](https://twitter.com/kamepon_fe)です。
```

**Zenn向け:**
```
こんにちは、[かめぽん](https://twitter.com/kamepon_fe)です。
```

**バリエーション例:**
- 「こんにちは、WebGLで飯が食いたい[かめぽん](https://twitter.com/kamepon_fe)です。」
- 「こんにちは、ビジュアルディベロッパー＆フロントエンドエンジニアをしています[かめぽん](https://twitter.com/kamepon_fe)です。」
- 「こんにちは、先日〇〇というニュースに驚きを隠せない、都内でフロントエンドエンジニアをしています[かめぽん](https://twitter.com/kamepon_fe)です。」

### 文体の特徴

1. **親しみやすい敬体（です・ます調）**
   - 「〜だと思います」「〜かと思います」
   - 「〜してみてください」「〜してみましょう」
   - 「〜ではないでしょうか」

2. **読者への呼びかけ**
   - 「みなさんご存知の方も多いと思いますが」
   - 「〜という方は〜してみるといいと思います」
   - 「ぜひ最後まで読んでみてください」

3. **個人的な感想・経験の共有**
   - 「僕は〜」「僕にとって〜」
   - 「個人的には〜」
   - 「〜と感じました」「〜と思っています」

4. **カジュアルな表現**
   - 「〜わけですが」「〜なんですが」
   - 「〜かもしれませんが」
   - 「めちゃくちゃ〜」「かなり〜」

### 締めくくりパターン

```
いかがだったでしょうか、[記事の要約]。
[今後の展望や読者へのメッセージ]。
最後まで読んでいただきありがとうございます！
```

または：

```
ここまで最後まで目を通していただいてありがとうございます。
[学びのポイントや感想]。
```

## 技術解説のパターン

### 比較説明パターン

```markdown
### [既存の方法/技術]の場合

[コード例]

こちらは[説明]ですね。しかし、[問題点や課題]という問題があります。

- [課題1]
- [課題2]
- [課題3]

### [新しい方法/技術]の場合

[コード例]

[新技術]では[特徴の説明]。[メリットの説明]することができます。
```

### 段階的な実装説明パターン

```markdown
## [セクション名]の準備

最初に[目的]をしますが、事前に[前提条件]を[行っておいてください/準備しておいてください]。

### [サブステップ1]

[説明]

[コード/コマンド例]

### [サブステップ2]

次に[次のステップの説明]。

[コード例]

この状態で[確認方法]。[期待する結果]が表示されたら成功です。
```

### アーキテクチャ説明パターン

```markdown
### アークテクチャ

![アーキテクチャ図](画像URL)

- [技術1]
- [技術2]
- [技術3]

今回使う構成は以上の通りで、[役割1]は[技術1]を使い、[役割2]は[技術2]、そして[役割3]で[技術3]を使います。[全体のフローの説明]という流れを作っていきます。
```

- JSON: `json`
- GraphQL: `graphql`
- シェル: `bash` または `sh`
- GLSL: `glsl`
- CSS: `css`
- HTML: `html`


## よく使うフレーズ集

### 導入部

- 「〜ということで、〜してみたいと思います」
- 「〜が気になっている人も多かと思いますが」
- 「この記事は割とボリュームが多いですが、一通りやれば〜の肌感は掴めると思うので」
- 「体系的な知識群として参考にしてもらえれば、と思います」

### 説明・解説

- 「〜と思ってもらって大丈夫です」
- 「イメージとしては〜のような感じです」
- 「端的にいうと〜」
- 「細かい仕様に関しては触れませんが」
- 「詳しくは以下を参照してみてください」

### 手順説明

- 「なにはともあれ、まず〜」
- 「早速〜を始めていきます」
- 「ここから〜の流れを説明していきます」
- 「ひとまず〜の設定は以上になります」
- 「以下のような画面が出ていれば成功です」

### 確認・成功

- 「〜が表示されたら成功です」
- 「〜になっていることが確認できれば〜成功しています」
- 「試しに〜してみます」
- 「〜で確認してみましょう」

### 注意喚起

- 「〜で一点注意があり、〜」
- 「注意点としては〜」
- 「ここで注意しなければいけないのは〜」

### 補足・追加情報

- 「ちなみに〜」
- 「もちろん〜」
- 「加えて〜」
- 「（もっとちゃんとやるのであれば〜）」

### 締め

- 「いかがだったでしょうか」
- 「〜ができたかと思います」
- 「〜として非常に魅力的なので今後少しづつ使っていければと思います」

## 記事タイプ別テンプレート

### チュートリアル記事

1. **概要**: 何を作るか、なぜ作るか
2. **ターゲット**: 誰向けの記事か（オプション）
3. **メリット**: この記事で得られること（オプション）
4. **アーキテクチャ/構成**: 全体像の説明
5. **環境構築**: 必要なセットアップ
6. **実装**: ステップバイステップの実装
7. **動作確認**: 完成の確認方法
8. **まとめ**: 学んだこと、感想
9. **参考**: 参考リンク

### 概念解説記事

1. **初めに/概要**: トピックの紹介
2. **〇〇とは**: 基本概念の説明
3. **なぜ〇〇が必要か**: 背景・課題
4. **〇〇の特徴/メリット**: 具体的な利点
5. **実際の使い方**: コード例
6. **注意点/ベストプラクティス**: 気をつけること
7. **まとめ**: 要点の整理
8. **参考**: 参考リンク

### ツール紹介記事

1. **初めに/概要**: ツールの概要
2. **〇〇とは**: ツールの説明と特徴
3. **導入方法**: インストール手順
4. **基本的な使い方**: 主要機能の紹介
5. **設定について**: カスタマイズ方法
6. **使ってみての感想**: 個人的な評価
7. **注意点**: 気をつけること
8. **参考**: 参考リンク

## チェックリスト

### 執筆前

- [ ] 記事のターゲット読者を明確にする
- [ ] 伝えたいポイントを3つ以内に絞る
- [ ] 必要なコードサンプルを準備する
- [ ] スクリーンショットを撮影する

### 執筆中

- [ ] 冒頭で自己紹介と記事の概要を述べる
- [ ] 見出しを適切に使い階層構造を作る
- [ ] コードブロックに言語/ファイル名を指定する
- [ ] 専門用語は初出時に説明を加える
- [ ] 長い説明は箇条書きで整理する

### 執筆後

- [ ] タイトルが検索されやすいか確認
- [ ] タグ/トピックスが適切か確認
- [ ] コードが実際に動作するか確認
- [ ] 参考リンクが有効か確認
- [ ] 誤字脱字がないか確認
- [ ] 画像が正しく表示されるか確認

## プラットフォーム別の注意点

### Qiita

- タグは5つまで設定可能
- 技術タグ + 「フロントエンド」の組み合わせが多い
- 画像はQiitaのストレージにアップロード
- Markdown記法はほぼ標準

### Zenn

- トピックスは複数設定可能
- emoji は記事のアイコンとして表示される
- URLを貼るとカード形式で表示される
- Zenn独自の記法（メッセージボックスなど）が使える
- 画像はZennのストレージにアップロード
